## 0. 검증의 기본 원칙 (멘토 피드백 반영)

1. **베이스라인부터 측정한다.** 기능을 더 얹기 전에, 현재 시스템의 자원·타이밍 특성을 먼저 확인해야
   "탭/그래프를 추가하면 무엇이 한계에 부딪히는가"를 알 수 있다.
2. **응답시간 = 실행시간(execution) + 대기시간(wait).** 종단간 한 숫자만 보지 말고 **단계로 분해**해서,
   "어디가 느린가(실행)"와 "부하 때문에 얼마나 기다리는가(대기)"를 따로 확인한다.
3. **Environment = 부하(load) 를 정의해야 한다.** 모든 타이밍 항목은 "시스템이 그때 무엇을 동시에 하고 있는가"
   (그래프 N개 렌더링, 샘플레이트, Pi 내부 IO 등)를 명시한 상태에서 측정해야 의미가 있다. → **normal load / peak load** 구분.
4. **목표값은 '평가 기준'에서 와야 한다.** 문서의 수치(특히 `(TBD)` 표기)는 현재 **가정값**이다.
   측정의 목적은 "달성됐나"보다 먼저 **"이 목표가 타당한가 / 확정·수정할 값은 무엇인가"** 를 판단하는 것.
5. 각 항목 **3회 이상 반복**, 첫 회(워밍업) 제외 고려, **평균 + 최악(worst/p95/p99)** 을 함께 본다.

각 항목 형식:
`📍 문서 위치` / `🎯 확인 목적(왜)` / `✅ 확인할 것(무엇을·어떤 조건)` / `📐 합격 기준`

## A. 응답 속도 / 지연 (Latency)

### A-1. 종단간 지연 (가장 중요)
- 📍 **QA-LT-01** (Response Measure: "캡처 시각 → 화면 표시 시각의 차이 평균 ≤50 ms · 최악 ≤100 ms"), **EXP-02**, Tracking Metrics
- 🎯 "Live Mode에서 실시간으로 시계 거동을 관찰 가능"하다는 **실시간성 주장**을 검증. 이 프로젝트 최상위 품질속성.
- ✅ 외부 비트가 마이크에 도달 → GUI에 해당 측정값이 보일 때까지의 시간을 **비트(이벤트) 단위**로 측정.
  블록 평균이 아니라 **각 A/C 이벤트의 캡처→표시 시간**을 모아 분포를 본다. 환경(부하)을 함께 기록.
- 📐 평균 **≤50 ms**, 최악 **≤100 ms**. (가정값 → 평가 기준 확보 시 재조정 가능)

### A-2. 지연의 단계 분해 (실행 vs 대기)
- 📍 **QA-LT-01 Plan 근거**("Low Latency 절 — 캡처→처리→표시 3단계 지연 정의"), **EXP-02**(단계별 병목 식별)
  ※ 원래 3단계 보고 요구였던 **FR-SYS-4가 현재 문서에서 빠져 있음** → 계측엔 여전히 필요(복구 권장).
- 🎯 "어디서 시간이 드는지" 규명. 멘토 핵심: 실행시간(태스크 처리)과 대기시간(부하·스케줄) 분리.
- ✅ **캡처→처리 지연**, **처리→표시 지연**, **종단간** 을 각각 측정 + **백로그(미처리 누적 샘플)** 관찰.
- 📐 (절대 임계 없음) — 각 단계의 평균/최악과 백로그 추세로 **병목 위치**를 식별하면 됨. 이후 queuing 모델 입력으로 사용.

### A-3. GUI 입력 응답성
- 📍 **QA-RT-01** ("GUI 입력 응답 시간 ≤200 ms"), **EXP-06**
- 🎯 측정·렌더 부하 중에도 **UI가 멈추지 않고 반응**하는지(반응형 GUI 주장) 검증.
- ✅ 부하 상태(Live + 다중 탭 렌더링)에서 버튼/입력 조작 → 반영까지 시간.
- 📐 **≤200 ms**.

### A-4. 결함 피드백 / 카운터 반영 지연
- 📍 **QA-AC-04**("오류 메시지 표시 ≤2초"), **QA-US-01**("결함 표시 ≤2초"), **QA-LT-03**("카운트 증가 5초 이내 화면 반영")
- 🎯 결함(신호 손실·과부하)을 사용자가 **적시에 인지**하는지(우아한 성능 저하·사용성) 검증.
- ✅ 결함/부하 주입 → 알림·카운터가 화면에 뜰 때까지 시간.
- 📐 알림 **≤2초**, 드롭/누락 카운터 화면 반영 **≤5초**.

## C. CPU · 발열 (연산 자원)

> ⚠️ **이 절(C)과 다음 절(D)의 자원 지표(CPU·메모리·스로틀)는 `perf_log.csv` 에 기록되지 않습니다.**
> 프로세스 안에서 측정하면 측정 행위 자체가 CPU·메모리를 먹어 결과를 오염시키므로(관측자 효과),
> **앱을 건드리지 않는 외부 도구로 측정**합니다. 앱 내부 계측은 밖에서 못 보는 의미론적 지표
> (지연 A·정확도 G·FPS F·백로그 E·이벤트루프 지연 A-3)만 담당합니다.
>
> **외부 측정 런북 (Raspberry Pi / Linux):**
>
> 메모리(D-1·D-2) + 발열(C-2) — **설치 불필요**, `/proc/smaps_rollup`+`vcgencmd` 합본 샘플러:
> ```bash
> tools/resource_sample.sh            # TimeGrapher 자동 탐색, 1초 주기 → resource_ext.csv
> tools/resource_sample.sh -i 2 -o run1.csv
> ```
> - **RSS 가 아니라 PSS 를 본다.** RSS 는 공유 라이브러리(Qt)를 통째로 세어 부풀려지고,
>   PSS 는 공유 페이지를 공유 프로세스 수로 나눠 배분 → 앱이 '진짜' 차지하는 몫. 절대 footprint는 PSS가 정확.
> - **누수(증가 추세)는 `Private_Dirty`** 가 가장 깨끗(앱 고유 힙). RSS는 CSV에 교차검증용으로 함께 기록.
> - CSV(합본): `epoch_s,pss_kb,pss_mb,private_dirty_kb,rss_kb,temp_c,arm_mhz,throttled_hex,now_throttling,ever_throttled`
>
> CPU(C-1) — 간단히는 `htop`(F4로 `TimeGrapher` 필터, `RES`/`CPU%` 열). 시계열·그래프가 필요하면
> 도구 설치 후: `pip install psrecord` → `psrecord $(pidof TimeGrapher) --interval 1 --plot perf_ext.png`,
> 또는 `sudo apt install sysstat` → `pidstat -r -u -p $(pidof TimeGrapher) 1`.
>
> 발열·스로틀(C-2) — 위 `resource_sample.sh` 가 발열도 함께 `resource_ext.csv` 에 기록한다(별도 실행 불필요).
> - `get_throttled` 비트: bit0~3=**현재 상태**(bit2=현재 스로틀), bit16~19=**부팅 후 발생 이력**(bit18=스로틀 이력).
>   `0x0` 이면 완전 정상. **이력 비트(0x..0000)는 재부팅 시 클리어** → 측정은 재부팅 후 0x0에서 시작 권장.
> - 빠른 단발 확인: `vcgencmd measure_temp` / `vcgencmd get_throttled`
>
> **외부 ↔ 내부 시간 정렬 (자동):**
> 내부 `t_ms` 는 단조시계(앱 시작=0), 외부 `epoch_s` 는 벽시계라 축이 다르다. `perf_log.csv` 헤더의
> `epoch_ms_t0`(= t_ms=0 의 벽시계 epoch[ms])가 변환 앵커다: `event_epoch_ms = epoch_ms_t0 + t_ms`.
> 이 변환을 자동으로 해 주는 도구:
> ```bash
> # 통합 롱포맷(epoch_ms,t_ms,source,metric,value)으로 합치기
> tools/perf_join.py build/perf_log.csv --mem mem_ext.csv --thermal thermal_ext.csv -o joined.csv
> # 내부 지표 이벤트마다 '그 시각의' 온도/메모리를 최근접 첨부 (예: 지연 스파이크가 발열과 겹쳤나)
> tools/perf_join.py build/perf_log.csv --thermal thermal_ext.csv --correlate e2e_full_ms --tolerance 1500 -o corr.csv
> ```
> (구버전 CSV에 `epoch_ms_t0` 가 없으면 정렬 불가 → 앵커 추가 후 재측정 필요. python 표준 라이브러리만 사용.)

- 📍 **QA-AC-01** ("T1·T3 검출률 ≥95% · FP ≤2%"), **EXP-03**
- 🎯 측정의 기반 이벤트(T1=rate/BE, T3=amplitude)를 **고신뢰로 검출**하는지.
- ✅ Sim 1,000비트(BPH 21,600~36,000, Amp 200~300°) vs ground-truth.
- 📐 검출률 **≥95%**, False Positive **≤2%**.

### G-3. 잡음 환경 강건성
- 📍 **QA-CO-04** ("검출률 ≥80% 유지 · 측정오차 증가 ≤2배"), **EXP-09**
- 🎯 60 dB 대화/진동 등 환경 잡음 하에서도 **검출·측정이 무너지지 않는지**.
- ✅ 무잡음 baseline 대비 잡음(60 dB SPL) 주입 후 검출률·오차 변화. (필터 효과 비교)
- 📐 검출률 **≥80%** 유지, 측정오차 증가 **≤2배**.

## K. 본 HW 성능 가이드에서 제외한 항목 (누락이 아님 — 성격이 다름)

> (검증은 하되 **기능/무결성/사용성/개발 지표** 트랙에서 별도로 확인.)

| **QA-SE-02** / EXP-12 저장 무결성 | Save→Load 체크섬 100% | **무결성** — 저장/로드 체크섬 검증 |
